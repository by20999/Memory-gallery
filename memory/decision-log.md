# 决策记录

## 2026-06-11 - 回忆相册定位与轻量前端体验

- 背景：用户希望相册不再只围绕“家庭”展开，而是面向更开放的回忆、关系、旅行、日常和灵感整理场景；同时希望界面减少毛玻璃卡片和 emoji 痕迹，靠排版、留白和克制控件提升质感。
- 决策：将项目中文定位调整为“共享回忆相册/回忆相册”；继续使用原生 HTML/CSS/ES Modules，不引入前端框架或组件库；通过重构现有 HTML/CSS 和少量原生模块改善体验。
- 影响：文案、README、项目总览、API 示例、数据模型示例和协作规范已同步更新；后续功能设计应避免只围绕家庭成员场景描述。
- 备选方案：引入 shadcn/React 或完整设计系统，但会提高构建和维护成本，不符合当前轻量项目边界。

## 2026-06-11 - 自定义头像采用浏览器本地存储

- 背景：用户希望增加自定义头像功能，但项目当前没有账号体系，也不希望引入多租户或复杂用户资料模型。
- 决策：头像图片在浏览器端裁剪压缩为小尺寸 data URL，保存到 `localStorage` 的 `album_avatar_image`；昵称继续保存在 `album_nickname`。头像不上传服务器，不写入 JSON 数据文件。
- 影响：实现简单、隐私风险低、不会改变后端数据模型；换设备、清理浏览器数据或换浏览器后需要重新设置头像。
- 备选方案：把头像上传到 `uploads/` 或新增用户资料 JSON，但会引入文件清理、权限和跨设备身份问题，暂不采用。

## 2026-06-11 - 原生 select 保留状态源，使用轻量增强层

- 背景：浏览器原生下拉菜单在当前视觉系统中显得粗糙，但现有业务逻辑依赖 `select.value` 和 `change` 事件。
- 决策：新增 `js/select.js`，保留原生 `select` 作为真实状态源，在旁边生成统一样式的触发器和菜单，并在选择后同步原 select 的值与 `change` 事件。
- 影响：上传整理、排序、内容筛选和分组选择视觉统一；业务逻辑基本不需要改动。
- 备选方案：完全替换为自定义状态组件，但会增加和现有模块同步的复杂度。

## 2026-06-11 - 每轮项目建设必须留痕

- 背景：项目已有较多文档和渐进式改动，如果每次对话建设后不记录，后续维护者难以理解设计取舍和当前状态。
- 决策：把“每轮项目建设、重要修复、UI/交互调整或功能新增后必须更新 `docs/project-log.md`”写入 `AGENTS.md` 和 `docs/development-constraints.md`；涉及长期取舍时同步写入 `memory/decision-log.md`。
- 影响：后续新对话结束时，应留下完成内容、关键决策、验证结果、风险和后续事项，避免只靠 Git diff 猜上下文。
- 备选方案：只在提交信息里说明，但本项目未强制每轮提交，且文档更方便非 Git 使用者查看。

## 2026-06-11 - 照片清晰度采用原图优先策略

- 背景：用户反馈上传后页面大图清晰度明显偏低，影响相册的回看体验。排查发现主画廊卡片和时间线封面存在使用 640px 缩略图撑大展示的情况，上传前也会默认压到较低上限。
- 决策：上传链路改为 100MB 内优先保留原图；超过上限的超大图片才在前端做高质量降采样。主画廊卡片、最近上传记忆卡和时间线封面改为原图优先，小封面、小缩略图继续使用 `thumbSrc`。
- 影响：照片大图观感更接近原始文件；服务器单张上传限制同步提升到 100MB；后端缩略图提升为 1600px/质量 92 的高质量预览，旧低清缩略图会在读取照片时自动重建；同屏大量高分辨率照片时资源压力高于纯缩略图方案，继续依赖懒加载和并发队列控制。
- 备选方案：所有列表继续使用缩略图，只在灯箱中显示原图。该方案更省流量，但用户在首页大图卡片上仍会感到发糊，不符合当前体验目标。

## 2026-06-12 - 故事模式与灯箱编辑页采用开放式工具界面

- 背景：故事模式左侧卡片占位过大，已有故事仍出现“创建第一本故事”的全局空状态；灯箱编辑页仍保留较多毛玻璃、胶囊按钮和卡片式信息块，和当前高级简约方向不一致。
- 决策：故事模式改为窄侧栏 + 开放式主内容区，已有故事优先展示当前故事详情和叙述编辑；灯箱编辑页改为暗色工具界面，编辑栏、滤镜栏、评论区和操作按钮尽量减少玻璃、阴影和过度圆角。全局补充 `[hidden] { display: none !important; }`，保证隐藏状态不被组件样式覆盖。
- 影响：故事列表按创建时间升序呈现，当前已有故事会自动选中；空故事提示只针对当前故事，减少误导；灯箱编辑页更偏专业查看/调整工具。
- 备选方案：继续保留卡片容器，只缩小尺寸。该方案改动更小，但仍会保留“AI 模板感”和首屏占位过大的问题，因此未采用。

## 2026-06-12 - 故事背景与全屏回忆剧场

- 背景：用户希望故事模式不要完全去卡片化导致失去层次，同时要求增加可自定义背景、透明度控制和更有氛围的全屏时间流线展示。
- 决策：故事对象新增 `backgroundPhotoId` 和 `backgroundOpacity`，用于持久化相册图片背景和透明度；电脑本地上传背景只保存在当前浏览器 `localStorage`，键名为 `album_story_background_<storyId>`，不写入服务器、不混入 `uploads/`。

- 备选方案：把本地背景作为照片上传到 `uploads/` 并写入 JSON。该方案会改变相册真实内容边界，也需要新增清理和权限规则，本轮未采用。

## 2026-06-15 - 故事全屏文案复用既有字段，位置仅本地保存（已被后续决策替代）

- 背景：用户希望故事全屏展示里的标题下方文案和底部寄语可以自行编辑、移动，同时不要再显示参考图中的占位文案；项目仍应保持轻量 JSON 存储和低迁移成本。
- 决策：标题下方文案复用故事 `description` 字段，底部寄语复用故事 `content` 字段；两个文案栏的拖动偏移保存到当前浏览器 `localStorage` 的 `album_story_showcase_text_layout_<storyId>`，不新增后端字段、不改变 `story-data.json` 结构。
- 影响：文案内容可以随故事数据备份和迁移，位置属于当前浏览器展示偏好，换设备后会回到默认位置；这符合当前无账号、轻量单机/小规模部署边界。
- 备选方案：在 `story-data.json` 新增全屏文案位置字段。该方案能跨设备同步位置，但会扩大数据模型并要求更多兼容处理，本轮暂不采用。
- 后续更新：用户明确要求普通“故事叙述”另作他用，因此本决策已被“全屏展示短文案从故事叙述中分离”替代；当前实现以 `showcaseSubtitle/showcaseFooter` 为准。

## 2026-06-13 - 故事全屏时间线继续使用原生 CSS/JS 氛围化实现

- 背景：用户希望全屏展示更有氛围：不规则圆形照片、上下交错、描述随位置展示、悬停轻微放大、曲线更蜿蜒和左右朦胧渐隐。
- 决策：继续用 `js/story.js` 计算时间线节点和 lane 类名，用 `style.css` 实现不规则圆框、hover 动效、边缘 mask 和字体调整，不引入动画库、Canvas 或新前端框架。
- 影响：全屏展示保持轻量、可维护，仍能复用当前故事数据和相册图片；CSS 覆盖层会继续承担较多视觉职责。
- 备选方案：引入 SVG/Canvas 专门绘制故事舞台。该方案可以更自由，但会扩大实现和调试成本，暂不符合轻量相册边界。

## 2026-06-15 - 故事全屏展示升级为代码生成的回忆海报视觉

- 背景：用户要求故事模式全屏展示尽可能贴近参考图，重点还原发光时间流线、圆形照片边框、星空粒子、樱花、胶片、纸张和二次元年轻用户喜欢的梦幻氛围；同时项目仍要求保持轻量、原生前端和低迁移成本。
- 决策：不引入数据库、构建系统、前端框架或大型组件库；使用 `index.html` 增加全屏装饰层，用 `js/story.js` 生成年份/标题/短句节点结构，用 `style.css` 末尾集中覆盖实现多层 SVG/CSS 发光、边角装饰、粒子、花瓣和响应式降级。
- 影响：全屏展示能在现有数据结构上获得更接近参考图的海报效果，不增加运行依赖，也不改变故事 JSON 字段；代价是 `style.css` 的故事全屏覆盖层继续变厚，需要后续有计划地拆分维护。
- 备选方案：下载或生成大量透明 PNG 装饰素材并作为静态资源引用。该方案视觉上可进一步逼近参考图，但会增加素材授权、体积、路径维护和部署迁移成本，本轮先采用代码生成装饰。
## 2026-06-15 - 故事全屏装饰允许使用可追溯本地素材

- 背景：用户明确指出纯 CSS/SVG 规则图形过于规整，缺少现实感，希望在可免费学习和下载的前提下使用网上素材，并要求记录所有来源。
- 决策：新增 `assets/story-decor/` 作为 UI 装饰素材目录，下载并引用 Openclipart 的樱花、胶片和纸张 SVG；素材只用于故事全屏边缘装饰，不放入 `uploads/`，避免与用户照片数据混淆。
- 影响：全屏展示边缘装饰比纯渐变/伪元素更自然；资源来源、下载链接、使用位置和效果统一记录在 `learning-plan/story-showcase-resources-2026-06-15.md`。FreeSVG 自动下载仍因 403 失败，只记录手动下载方法，不声称已使用。
- 备选方案：继续完全用 CSS 生成装饰，维护成本更低但规整感明显；或引入大量 PNG/插画素材，视觉可能更接近参考图但授权、体积和迁移成本更高，本轮先采用少量可追溯 SVG。

## 2026-06-15 - 全屏展示短文案从故事叙述中分离

- 背景：用户指出普通“故事叙述”窗口有其他用途，不应被全屏标题下方文案和底部寄语占用；全屏展示文案需要独立入口并限制字数。
- 决策：故事对象新增 `showcaseSubtitle` 和 `showcaseFooter` 字段，分别存储全屏标题下方短文案和底部寄语；普通故事页在标题卡和背景卡中提供输入框和字数计数，`content` 继续只作为故事叙述长文本。
- 影响：全屏展示文案可以随 `story-data.json` 备份迁移，普通故事叙述不再被覆盖；全屏内直接编辑仍可微调这两个短字段，拖动位置继续保存在本地 `localStorage`。
- 备选方案：继续复用 `description/content`。该方案字段更少，但会混淆故事摘要、长叙述和海报短文案用途，因此不再采用。

## 2026-06-15 - 故事背景透明度尊重用户原图

- 背景：用户反馈全屏展示滤镜和光效太重，上传背景图片的色彩与内容被遮盖，而且透明度滑条在全屏里看起来不生效。
- 决策：`backgroundOpacity` 规范范围从 `0-0.55` 调整为 `0-1`；全屏展示不再强制提高背景不透明度，也不再给背景图套用重 blur、luminosity 或灰暗滤镜。
- 影响：100% 透明度时背景图会尽量按原始色彩展示；普通故事页和全屏展示共用同一个背景透明度字段，备份迁移方式不变。
- 备选方案：继续保留强氛围滤镜。该方案更接近海报合成，但会牺牲用户自选背景图的可辨识度，因此不再采用。

## 2026-06-15 - 全屏展示速度与装饰素材继续按小相册体验收敛

- 背景：用户反馈全屏自动播放最低速度仍过快，且参考图氛围不能只靠单一樱花素材，应尽量使用已放入 `assets/story-decor/` 的其他 SVG。
- 决策：自动播放速度范围收敛为 `30-72px/s`，默认 `40px/s`，滑条数值直接等于实际像素速度；全屏边缘装饰不再使用左上角和右下角的大花，改为保留低透明度胶片、纸张、拍立得、星尘、粒子和花瓣等轻细节。
- 影响：默认播放更舒缓，速度上限不会过快；装饰层更接近参考图的复合海报感，同时仍不新增前端框架或运行依赖。
- 备选方案：继续使用上一轮 `72-180px/s` 和单一主樱花素材。该方案更容易看出自动播放在移动，但不符合用户对慢速和素材丰富度的反馈，因此不采用。
## 2026-06-15 - 故事全屏展示改为应用内沉浸式遮罩（已被后续决策覆盖）

- 背景：用户反馈打开故事“全屏展示”时浏览器会显示原生全屏提示，这类提示属于浏览器安全 UI，页面无法通过 CSS 或 JavaScript 隐藏。
- 决策：故事展示不再调用 `requestFullscreen()` / `exitFullscreen()`，而是仅使用 `.story-showcase` 固定遮罩和 `html/body.story-showcase-open` 滚动锁定实现全视口沉浸展示。
- 影响：打开展示时不会再触发 Chrome 等浏览器的原生全屏提示；关闭按钮和 Esc 仍然关闭应用内展示层，自动播放、拖拽和全屏文案编辑继续保留。
- 备选方案：继续调用浏览器原生全屏 API 并尝试用样式遮挡提示。该方案不可行，因为原生提示不在页面 DOM 内，不能被项目样式控制。

## 2026-06-15 - 故事全屏展示恢复浏览器原生全屏

- 背景：用户确认相比隐藏浏览器全屏提示，更需要故事展示真正占满整个屏幕，而不是只覆盖网页视口。
- 决策：恢复 `requestFullscreen()`、`exitFullscreen()` 和 `fullscreenchange` 监听；故事展示打开时进入浏览器原生全屏，关闭按钮、Esc 或浏览器退出全屏都会同步关闭展示层。
- 影响：展示能真正进入系统级全屏；浏览器自带的 Esc 退出提示会保留，这是原生全屏 API 的安全提示，项目不再尝试规避。
- 同步调整：自动播放速度范围改为 `40-72px/s`，默认 `50px/s`，比上一轮默认 `40px/s` 更明显。

## 2026-06-15 - 故事全屏自动播放速度改为快速响应

- 背景：用户继续反馈默认自动播放仍偏慢，且拖动速度滑条后实际播放体感没有及时跟随，尤其大幅调整时需要重新进入全屏才明显生效。
- 决策：自动播放速度范围调整为 `50-72px/s`，默认 `54px/s`；播放中调整速度时立即取消旧动画帧、重置帧时间并重新排帧，让滑条数值直接对应下一帧后的滚动速度。
- 影响：全屏展示保留原生全屏提示与应用内展示层，自动播放速度的 UI 初始值、JS 默认值和实时播放行为保持一致。
