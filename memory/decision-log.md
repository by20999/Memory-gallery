# 决策记录

## 2026-06-11 - 回忆相册定位与轻量前端体验

- 背景：用户希望相册不再只围绕“家庭”展开，而是面向更开放的回忆、关系、旅行、日常和灵感整理场景；同时希望界面减少毛玻璃卡片和 emoji 痕迹，靠排版、留白和克制控件提升质感。
- 决策：将项目中文定位调整为“共享回忆相册/回忆相册”；继续使用原生 HTML/CSS/ES Modules，不引入前端框架或组件库；通过重构现有 HTML/CSS 和少量原生模块改善体验。
- 影响：文案、README、项目总览、API 示例、数据模型示例和协作规范已同步更新；后续功能设计应避免只围绕家庭成员场景描述。
- 备选方案：引入 shadcn/React 或完整设计系统，但会提高构建和维护成本，不符合当前轻量项目边界。

## 2026-06-11 - 自定义头像采用浏览器本地存储

- 背景：用户希望增加自定义头像功能，但项目当前没有账号体系，也不希望引入多租户或复杂用户资料模型。
- 决策：头像图片在浏览器端裁剪压缩为小尺寸 data URL，保存到 `localStorage` 的 `album_avatar_image`；昵称继续保存在 `album_nickname`。头像不上传服务器，不写入 JSON 数据文件。
- 影响：实现简单、隐私风险低、不会改变后端数据模型；换设备、清理浏览器数据或换浏览器后需要重新设置头像。
- 备选方案：把头像上传到 `uploads/` 或新增用户资料 JSON，但会引入文件清理、权限和跨设备身份问题，暂不采用。

## 2026-06-11 - 原生 select 保留状态源，使用轻量增强层

- 背景：浏览器原生下拉菜单在当前视觉系统中显得粗糙，但现有业务逻辑依赖 `select.value` 和 `change` 事件。
- 决策：新增 `js/select.js`，保留原生 `select` 作为真实状态源，在旁边生成统一样式的触发器和菜单，并在选择后同步原 select 的值与 `change` 事件。
- 影响：上传整理、排序、内容筛选和分组选择视觉统一；业务逻辑基本不需要改动。
- 备选方案：完全替换为自定义状态组件，但会增加和现有模块同步的复杂度。

## 2026-06-11 - 每轮项目建设必须留痕

- 背景：项目已有较多文档和渐进式改动，如果每次对话建设后不记录，后续维护者难以理解设计取舍和当前状态。
- 决策：把“每轮项目建设、重要修复、UI/交互调整或功能新增后必须更新 `docs/project-log.md`”写入 `AGENTS.md` 和 `docs/development-constraints.md`；涉及长期取舍时同步写入 `memory/decision-log.md`。
- 影响：后续新对话结束时，应留下完成内容、关键决策、验证结果、风险和后续事项，避免只靠 Git diff 猜上下文。
- 备选方案：只在提交信息里说明，但本项目未强制每轮提交，且文档更方便非 Git 使用者查看。

## 2026-06-11 - 照片清晰度采用原图优先策略

- 背景：用户反馈上传后页面大图清晰度明显偏低，影响相册的回看体验。排查发现主画廊卡片和时间线封面存在使用 640px 缩略图撑大展示的情况，上传前也会默认压到较低上限。
- 决策：上传链路改为 100MB 内优先保留原图；超过上限的超大图片才在前端做高质量降采样。主画廊卡片、最近上传记忆卡和时间线封面改为原图优先，小封面、小缩略图继续使用 `thumbSrc`。
- 影响：照片大图观感更接近原始文件；服务器单张上传限制同步提升到 100MB；后端缩略图提升为 1600px/质量 92 的高质量预览，旧低清缩略图会在读取照片时自动重建；同屏大量高分辨率照片时资源压力高于纯缩略图方案，继续依赖懒加载和并发队列控制。
- 备选方案：所有列表继续使用缩略图，只在灯箱中显示原图。该方案更省流量，但用户在首页大图卡片上仍会感到发糊，不符合当前体验目标。
