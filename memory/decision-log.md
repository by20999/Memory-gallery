# 决策记录

## 2026-06-11 - 回忆相册定位与轻量前端体验

- 背景：用户希望相册不再只围绕“家庭”展开，而是面向更开放的回忆、关系、旅行、日常和灵感整理场景；同时希望界面减少毛玻璃卡片和 emoji 痕迹，靠排版、留白和克制控件提升质感。
- 决策：将项目中文定位调整为“共享回忆相册/回忆相册”；继续使用原生 HTML/CSS/ES Modules，不引入前端框架或组件库；通过重构现有 HTML/CSS 和少量原生模块改善体验。
- 影响：文案、README、项目总览、API 示例、数据模型示例和协作规范已同步更新；后续功能设计应避免只围绕家庭成员场景描述。
- 备选方案：引入 shadcn/React 或完整设计系统，但会提高构建和维护成本，不符合当前轻量项目边界。

## 2026-06-11 - 自定义头像采用浏览器本地存储

- 背景：用户希望增加自定义头像功能，但项目当前没有账号体系，也不希望引入多租户或复杂用户资料模型。
- 决策：头像图片在浏览器端裁剪压缩为小尺寸 data URL，保存到 `localStorage` 的 `album_avatar_image`；昵称继续保存在 `album_nickname`。头像不上传服务器，不写入 JSON 数据文件。
- 影响：实现简单、隐私风险低、不会改变后端数据模型；换设备、清理浏览器数据或换浏览器后需要重新设置头像。
- 备选方案：把头像上传到 `uploads/` 或新增用户资料 JSON，但会引入文件清理、权限和跨设备身份问题，暂不采用。

## 2026-06-11 - 原生 select 保留状态源，使用轻量增强层

- 背景：浏览器原生下拉菜单在当前视觉系统中显得粗糙，但现有业务逻辑依赖 `select.value` 和 `change` 事件。
- 决策：新增 `js/select.js`，保留原生 `select` 作为真实状态源，在旁边生成统一样式的触发器和菜单，并在选择后同步原 select 的值与 `change` 事件。
- 影响：上传整理、排序、内容筛选和分组选择视觉统一；业务逻辑基本不需要改动。
- 备选方案：完全替换为自定义状态组件，但会增加和现有模块同步的复杂度。

## 2026-06-11 - 每轮项目建设必须留痕

- 背景：项目已有较多文档和渐进式改动，如果每次对话建设后不记录，后续维护者难以理解设计取舍和当前状态。
- 决策：把“每轮项目建设、重要修复、UI/交互调整或功能新增后必须更新 `docs/project-log.md`”写入 `AGENTS.md` 和 `docs/development-constraints.md`；涉及长期取舍时同步写入 `memory/decision-log.md`。
- 影响：后续新对话结束时，应留下完成内容、关键决策、验证结果、风险和后续事项，避免只靠 Git diff 猜上下文。
- 备选方案：只在提交信息里说明，但本项目未强制每轮提交，且文档更方便非 Git 使用者查看。

## 2026-06-11 - 照片清晰度采用原图优先策略

- 背景：用户反馈上传后页面大图清晰度明显偏低，影响相册的回看体验。排查发现主画廊卡片和时间线封面存在使用 640px 缩略图撑大展示的情况，上传前也会默认压到较低上限。
- 决策：上传链路改为 100MB 内优先保留原图；超过上限的超大图片才在前端做高质量降采样。主画廊卡片、最近上传记忆卡和时间线封面改为原图优先，小封面、小缩略图继续使用 `thumbSrc`。
- 影响：照片大图观感更接近原始文件；服务器单张上传限制同步提升到 100MB；后端缩略图提升为 1600px/质量 92 的高质量预览，旧低清缩略图会在读取照片时自动重建；同屏大量高分辨率照片时资源压力高于纯缩略图方案，继续依赖懒加载和并发队列控制。
- 备选方案：所有列表继续使用缩略图，只在灯箱中显示原图。该方案更省流量，但用户在首页大图卡片上仍会感到发糊，不符合当前体验目标。

## 2026-06-12 - 故事模式与灯箱编辑页采用开放式工具界面

- 背景：故事模式左侧卡片占位过大，已有故事仍出现“创建第一本故事”的全局空状态；灯箱编辑页仍保留较多毛玻璃、胶囊按钮和卡片式信息块，和当前高级简约方向不一致。
- 决策：故事模式改为窄侧栏 + 开放式主内容区，已有故事优先展示当前故事详情和叙述编辑；灯箱编辑页改为暗色工具界面，编辑栏、滤镜栏、评论区和操作按钮尽量减少玻璃、阴影和过度圆角。全局补充 `[hidden] { display: none !important; }`，保证隐藏状态不被组件样式覆盖。
- 影响：故事列表按创建时间升序呈现，当前已有故事会自动选中；空故事提示只针对当前故事，减少误导；灯箱编辑页更偏专业查看/调整工具。
- 备选方案：继续保留卡片容器，只缩小尺寸。该方案改动更小，但仍会保留“AI 模板感”和首屏占位过大的问题，因此未采用。

## 2026-06-12 - 故事背景与全屏回忆剧场

- 背景：用户希望故事模式不要完全去卡片化导致失去层次，同时要求增加可自定义背景、透明度控制和更有氛围的全屏时间流线展示。
- 决策：故事对象新增 `backgroundPhotoId` 和 `backgroundOpacity`，用于持久化相册图片背景和透明度；电脑本地上传背景只保存在当前浏览器 `localStorage`，键名为 `album_story_background_<storyId>`，不写入服务器、不混入 `uploads/`。
- 影响：故事模式普通编辑态和全屏展示态可以共用背景氛围；迁移服务器时，相册图片背景会随 JSON 迁移，本地上传背景不会跨浏览器同步。
- 备选方案：把本地背景作为照片上传到 `uploads/` 并写入 JSON。该方案会改变相册真实内容边界，也需要新增清理和权限规则，本轮未采用。
## 2026-06-13 - 参考图精修阶段继续采用局部覆盖层收口

- 背景：用户要求继续贴近参考图，并修正顶部栏、分页、故事模式布局和主题文案等一组体验问题；当前 `style.css` 历史层叠较多，但业务功能已稳定，且工作区包含多轮未提交改动。
- 决策：本轮继续延续原生 HTML/CSS/ES Modules，不引入框架或构建系统；视觉修正主要通过 `style.css` 末尾明确命名的覆盖块收口，JS 只做分页数量、主题文案事件和故事节点文本展示等必要调整。
- 影响：能在不重写大量历史样式的前提下快速修正用户可见问题，降低误伤上传、故事、分页、主题切换等既有功能的风险；代价是 CSS 覆盖层继续增加，后续维护需要更仔细确认优先级。
- 备选方案：立即拆分 shell、album、story 样式并清理旧规则。该方案长期更干净，但本轮会扩大改动面，容易影响现有真实相册数据下的交互稳定性，因此暂不采用。

## 2026-06-13 - 故事模式参考图还原只采用布局不采用配色

- 背景：用户明确指出参考图只用于还原故事模式排版布局，当前应用应继续使用夏日主题颜色，而不是切成参考图的米杏主题。
- 决策：故事模式排版改为参考图式的“左侧紧凑故事面板 + 右侧标题/背景 + 下方全宽时间线”，同时 story 面板、按钮、时间线和文本颜色统一读取 `--shell-*` 主题变量。
- 影响：夏日主题下故事模式保持蓝白清爽视觉，切换奶油、胶片或自定义主题时仍会跟随主题变量；参考图的结构被保留，但不再写死暖色纸面变量。
- 备选方案：直接照搬参考图的颜色与纸感。该方案与用户“主题颜色不要变”的要求冲突，因此不采用。

## 2026-06-13 - 全屏展示头部避免被故事模式全局隐藏规则误伤

- 背景：故事模式为了隐藏主相册页头使用了 `.story-view-active header { display: none; }`，但全屏展示内部也使用 `header.story-showcase-head`，导致右上角“功能”按钮随父级一起不可见。
- 决策：保留隐藏主页面 header 的规则，同时为 `.story-view-active .story-showcase-head` 增加精确覆盖，恢复全屏展示头部布局，并将功能按钮固定到全屏展示右上角。
- 影响：故事模式普通页面仍隐藏主相册标题，全屏展示的功能菜单恢复可见；后续新增全屏/弹层 header 时应避免被全局 `header` 选择器误伤。
- 备选方案：把全屏展示 HTML 的 `header` 标签改成 `div`。该方案会改动结构和语义，本轮用样式精确覆盖即可解决，因此暂不采用。

## 2026-06-13 - 记忆区和故事卡片继续以末尾覆盖块精修

- 背景：用户指出首页最近新增占位过大、故事模式左侧卡片被视觉拆分、故事背景下拉菜单被预览区覆盖；当前 `style.css` 已有多轮未提交覆盖层。
- 决策：不改 HTML 结构和故事数据模型，仅拆分最近新增/往年今日数量常量并统一限制为 3 张，在 `style.css` 末尾追加集中覆盖块修正布局宽度、左侧故事卡片层次和下拉菜单 z-index。
- 影响：能快速修复用户可见问题，避免重构旧样式误伤相册上传、分页和故事交互；代价是 CSS 层叠继续变厚，后续维护要优先查看文件末尾覆盖块。
- 备选方案：重构故事模式和记忆区样式为独立文件。该方案长期更清晰，但本轮改动面过大，暂不采用。

## 2026-06-13 - 故事全屏时间线继续使用原生 CSS/JS 氛围化实现

- 背景：用户希望全屏展示更有氛围：不规则圆形照片、上下交错、描述随位置展示、悬停轻微放大、曲线更蜿蜒和左右朦胧渐隐。
- 决策：继续用 `js/story.js` 计算时间线节点和 lane 类名，用 `style.css` 实现不规则圆框、hover 动效、边缘 mask 和字体调整，不引入动画库、Canvas 或新前端框架。
- 影响：全屏展示保持轻量、可维护，仍能复用当前故事数据和相册图片；CSS 覆盖层会继续承担较多视觉职责。
- 备选方案：引入 SVG/Canvas 专门绘制故事舞台。该方案可以更自由，但会扩大实现和调试成本，暂不符合轻量相册边界。
